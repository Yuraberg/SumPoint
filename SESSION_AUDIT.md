# Аудит сессий Claude Code — что стабильно съедает время

Дата аудита: 2026-07-07. Режим: только анализ, код не менялся.

**Источники сигнала.** Сырые расшифровки прошлых сессий недоступны (облачные сессии эфемерны), поэтому анализ построен на их следах: полная git-история (68 коммитов, 2026-04-14 → 2026-07-05, ~17 рабочих сессий), PR/ветки/CI-прогоны на GitHub (2 PR, 23 прогона CI, 5 прогонов Deploy), эволюция CLAUDE.md и BACKLOG.md, текущее состояние репозитория. Анализ выполнен тремя под-агентами: git-история, GitHub PR/CI, операционные цепочки.

Легенда решений: **skill** — оформить как skill/чеклист для Claude Code; **автоматизация** — CI/скрипты/hooks; **фикс** — точечная правка кода или доков; **не трогать** — не стоит усилий.

## P1 — максимальный возврат на вложенное время

### Группа 1. Сага MissingGreenlet: async SQLAlchemy × Celery — самый дорогой повторяющийся цикл
- **Сигнал:** одна и та же ошибка чинилась **8 раз тремя сессиями** за 15 дней, каждый раз новой стратегией с откатом предыдущей:
  - 2026-06-15 (4 попытки за 1ч12м): `4ddac1a` fresh event loop → `3fcc786` lazy engine per task → `e52b020` dispose в новом loop → `10f7372` не закрывать, GC разберётся;
  - 2026-06-28 (3 попытки за 41м): `022c005` await_fallback() → `7a3df1b` `--pool=solo` (threads ломает await_fallback) → `b094b56` NullPool;
  - 2026-06-30: `04d39ed` — откат await_fallback обратно на `asyncio.run()`.
  - CI при этом всегда зелёный — тесты эту зону не покрывают, всё выяснялось на проде.
- **Почему дорого:** классическая борьба с фундаментальной несовместимостью методом перебора; каждая новая сессия не знала, какие ветки уже тупиковые, и «чинила» заново.
- **Решение: фикс (два уровня).**
  1. *Дёшево, сразу:* зафиксировать в CLAUDE.md «золотой» паттерн async-в-Celery для этого проекта (`asyncio.run()` + `--pool=solo` + `NullPool`) **и список тупиковых веток с причинами**: `await_fallback()` — нет, `--pool=threads` — нет, ручной event loop — нет, dispose в отдельном loop — нет. Это запрет на четвёртый круг.
  2. *Правильно, при случае:* архитектурно закрыть класс ошибок — в Celery-тасках использовать синхронный engine (psycopg) вместо asyncpg-акробатики. Кандидат-файл: `app/tasks/base.py` (шаблон `_run()`).
- **Приоритет:** P1, самая высокая экономия.

### Группа 2. Инфраструктура прода живёт вне репозитория и переобъясняется каждую сессию
- **Сигнал:**
  - Реальный прод описан не в `docker-compose.yml` репо, а в `/root/vps-new/docker-compose.yml` (`dc2.yml`) на VPS; имена сервисов расходятся (`api/bot/worker/beat` в репо vs `sumpoint-api/...` в `deploy.yml:46-50`).
  - Ручной скрипт `/usr/local/bin/sumpoint-network-fix` после деплоя (BACKLOG P2, строка 28) — ручной шаг на каждый деплой.
  - Порт-флип-флоп за 12 минут: `81019aa` → revert `9b79698` (8000 конфликтует с Coolify nginx); Traefik file-provider как обход бага Coolify 4.1.2 (BACKLOG:51).
  - Ollama: три разных адреса (config `172.20.0.1`, `.env.example` localhost, прод — не записан нигде) + тихий fallback на zero-vector, молча деградирующий семантический поиск (`app/services/ai_engine.py:131-142`).
  - Настройка автодеплоя 2026-07-05: 5 прогонов за день, классические грабли (`git pull` без стратегии на разошедшейся истории, неверный путь к `dc2.yml`), 3 правки скрипта — и всё равно финальный run упал.
  - Правки файлов прямо на VPS ломали деплой: `984fa82` «restore **corrupted** config.py redis_url and truncated docker-compose env vars»; позже `e0c878e` ввёл `reset --hard` именно из-за «local edits on VPS».
- **Решение: skill + автоматизация + фикс.**
  1. **Skill `deploy-context`** (главное): единый документ-скилл с топологией прода — пути `/root/vps-new`, имена `sumpoint-*`, роли Coolify/Caddy/Traefik file-provider, где живёт Ollama и его прод-адрес, зачем `sumpoint-network-fix`, как смотреть логи/health, правило «порт = 8001:8000, НЕ 8000», правило «на VPS ничего не редактировать руками — только git».
  2. **Автоматизация:** закрыть BACKLOG-пункт про auto-network-fix (вкрутить в compose/деплой-хук); retry в `deploy.yml` на `docker compose up -d` (или `rm -f` перед up) — лечит гонку из P0; алерт на упавший deploy run.
  3. **Фикс:** заменить тихий zero-vector fallback на громкий (лог уровня error + не писать zero-vector в БД или помечать пост).
- **Приоритет:** P1.

### Группа 3. Документация врёт или молчит — источник «переобъяснения контекста»
- **Сигнал:**
  - README до сих пор утверждает, что AI-пайплайн — «Claude (Opus 4.6)» и требует `ANTHROPIC_API_KEY` (`README.md:32,62,137`), тогда как код ходит в DeepSeek + Ollama. Каждая сессия перепроверяет, «на чём реально крутится AI».
  - В CLAUDE.md отсутствуют целые подсистемы, которые есть в коде: schedule-подсистема (`app/api/schedule.py`, `app/tasks/schedule_tasks.py`), keyword-alerts, magic-link login, rate-limiting за Caddy (`app/rate_limit.py:8-11` — все юзеры за одним IP).
  - Маятниковые решения между сессиями — сессия N молча отменяет решение сессии N-1: nightly fetch (`b47eb6c`) → возврат continuous (`a9d0a30`) за 2 дня; утечка бот-токена в логах чинилась дважды (`68c6be5` 19.06 и `8ea57ea` 05.07); переименования моделей Fast/Reasoner → Flash/Pro → V4.
- **Решение: фикс (дёшево, высокий ROI).**
  1. Привести README к реальности (DeepSeek + Ollama, убрать ANTHROPIC_API_KEY).
  2. Дописать в CLAUDE.md: schedule, keyword-alerts, magic-link, rate-limit-за-Caddy; константы, которые переоткрывались: порт `8001:8000`, `vector(1024)`/BGE-M3, «везде naive UTC через `utcnow()`-хелпер», «никаких fire-and-forget `create_task` в хендлерах».
  3. Завести раздел **DECISIONS** (в CLAUDE.md или BACKLOG): «почему так и не менять без причины X» — страховка от маятника.
- **Приоритет:** P1.

## P2 — полезно, делать по мере надобности

### Группа 4. Регенерация Telethon-сессии — повторяющийся ручной ритуал
- **Сигнал:** интерактивный `generate_session.py` (phone + код из Telegram) → руками вставить `TELEGRAM_SESSION_STRING` в Coolify → рестарт worker → руками выяснить, какие каналы отвалились. В BACKLOG уже висит P1 «Обновить Telethon-сессию» с вручную собранным списком из 4 каналов «Could not find input entity».
- **Решение: skill `refresh-telethon-session`.** Пошаговый чеклист (полная автоматизация невозможна — интерактивный код из Telegram). Дополнительно фикс: команда/эндпоинт «показать каналы с `last_error`», чтобы диагностика не собиралась руками.

### Группа 5. Локальный запуск и отладка вслепую на проде
- **Сигнал:** полное окружение = 5 процессов в 5 терминалах + миграции + подводные камни (`--pool=solo`, Telethon только в worker). Следствие — отладка на проде: 2026-06-01, 9 коммитов за день, включая два подряд «add error diagnostics» в задеплоенный сервис (`3325313`, `8464359`) — ошибка не воспроизводилась локально.
- **Решение: skill `dev-up` (лёгкий).** Готовая последовательность команд с граблями. Альтернатива — Makefile, но skill дешевле и не трогает репо-код.

### Группа 6. Пробелы CI и тестов вокруг «выстраданных» мест
- **Сигнал:**
  - Нет lint-шага, нет `alembic check`/прогона миграций на живой БД (в CI `DATABASE_URL` указывает на несуществующий Postgres — `ci.yml:16`); блок тестов задублирован один-в-один в `ci.yml` и `deploy.yml`.
  - 51 тест покрывает в основном чистые утилиты; НЕ покрыты именно многократно чинившиеся зоны: dedup-savepoint, Celery-таски (MissingGreenlet), broker-доставка (тихий отказ на месяц: `a9d0a30` «no posts/digests were ever delivered»), bot-хендлеры (тихие падения: `d60b11b`, `f3df409`, `9911548`).
- **Решение: автоматизация.** (а) lint-шаг; (б) Postgres+pgvector как service-контейнер + `alembic upgrade head` в CI; (в) reusable workflow вместо дубля; (г) 2-3 интеграционных теста на dedup и broker-smoke («таск реально проходит через Redis»). Обновить deprecated actions (checkout@v4/setup-python@v5 на Node 20) — заодно.

### Группа 7. Гигиена веток
- **Сигнал:** одна вечная ветка `claude/wonderful-curie-1qmtew` использовалась для PR #1, PR #2 и рефакторинга 04.07 → ручной merge main с конфликтами в 7 файлах (`1b2e0b7`); ветка-зомби жива до сих пор при уже влитом содержимом.
- **Решение: автоматизация (настройка, не код).** Включить в GitHub «Automatically delete head branches»; привычка — свежая ветка на сессию.
